# МИНИСТЕРСТВО ОБРАЗОВАНИЯ И НАУКИ
## Модуль 1. Анализ требований, проектирование и разработка программных продуктов
### Пояснительная записка к проекту автоматизированной информационной системы «Интернет-продажи»
### Пункт 1. Техническое задание на создание программного продукта «Интернет-продажи»
1.1. Общие сведения и основание для разработки
Настоящее техническое задание определяет требования к разработке автоматизированной информационной системы (АИС) «Интернет-продажи». Проектирование и разработка архитектуры системы выполняются в соответствии с основными положениями ГОСТ 34.602-2020. Программа создается как самостоятельная конфигурация на базе современной технологической платформы «1С:Предприятие 8.3».
1.2. Назначение и цели создания системы
Основным назначением разрабатываемого программного продукта является автоматизация сквозного учета торговой деятельности сети интернет-магазинов, реализующих бытовую технику. Система предназначена для диспетчеров, менеджеров по продажам и курьерской службы.
Внедрение системы преследует следующие цели:
 * Анализ текущего потребительского спроса на конкретные модели товаров от различных мировых производителей.
 * Отслеживание динамики реализации бытовой техники за произвольные интервалы времени (дни, недели, месяцы).
 * Мониторинг, сравнение и контроль условий работы и качества доставки товаров до конечного потребителя в различных интернет-магазинах.
1.3. Функциональные требования к системе
Программный продукт должен обеспечивать круглосуточную регистрацию поступающих заявок, хранение нормативно-справочной информации о структуре торговой сети, ведение каталога товаров, фиксацию факта телефонного подтверждения заказов, а также автоматическое распределение поручений на доставку между курьерами.
### Пункт 2 и 3. Проектирование базы данных и представление объектов в виде таблиц
В рамках проектирования структуры базы данных на платформе «1С:Предприятие 8.3» было принято проектное решение отказаться от сложных и громоздких промежуточных таблиц. Для упрощения логики работы системы и минимизации рисков возникновения ошибок при вводе данных установлено допущение: один документ «Заказ» оформляется на одну конкретную позицию товара в определенном количестве. Данный подход позволяет линейно анализировать спрос и динамику без усложнения структуры метаданных.
Ниже представлено текстовое описание базовых таблиц (объектов метаданных), состава их полей и типов данных.
1. Подсистема «ЗакупкиИПродажи»
Данная подсистема отвечает за формирование пользовательского интерфейса, группируя основные справочники и документы, связанные с продажами.
 * Подсистема «ДоставкаТоваров»
   Данная подсистема организует интерфейс для логистов и курьеров, собирая объекты, отвечающие за транспортировку заказов.
2. Перечисление «ВидыТоваров»
Используется как фиксированный список для классификации бытовой техники. Значения задаются в конфигураторе и не могут быть изменены пользователем.
 * Значения перечисления: Утюг, Электрочайник, МикроволноваяПечь, Пылесос, Блендер, Фен, СтиральнаяМашина, Холодильник, Тостер.
 * Свойства: Длина имени стандартная, индексация включена по умолчанию. Связано с подсистемой «ЗакупкиИПродажи».
3. Справочник «Фирмы»
Предназначен для хранения списка заводов-производителей и брендов бытовой техники.
 * Поле «Наименование» (Стандартный реквизит). Тип данных: Строка, длина 50 символов. Является ключевым полем для поиска. Не заполнено быть не может.
 * Свойства: Код уникальный, генерируется автоматически. Связано с подсистемой «ЗакупкиИПродажи».
4. Справочник «Магазины»
Хранит информацию обо всех торговых точках и интернет-площадках, входящих в общую базу.
 * Поле «Наименование» (Стандартный реквизит). Тип данных: Строка, длина 100 символов. Является ключевым текстовым полем.
 * Свойства: Автонумерация кодов включена.

Связано с подсистемой «ЗакупкиИПродажи».
5. Справочник «Товары»
Основной каталог продаваемой бытовой техники.
 * Поле «Наименование» (Стандартный реквизит). Тип данных: Строка, длина 100 символов. Содержит конкретное название модели.
 * Поле «ВидТовара» (Реквизит). Тип данных: ПеречислениеСсылка.ВидыТоваров. Обязательное для заполнения поле.
 * Поле «ФирмаПроизводитель» (Реквизит). Тип данных: СправочникСсылка.Фирмы. Внешний ключ для связи с брендами.
 * Поле «Цена» (Реквизит). Тип данных: Число, длина 10 знаков, точность 2 знака после запятой. Не может быть отрицательным.
 * Свойства: Уникальный код товара. Связано с подсистемой «ЗакупкиИПродажи».
6. Документ «ЗаказПокупателя»
Фиксирует событие обращения клиента и покупку товара.
 * Поле «Номер» и «Дата» (Стандартные реквизиты). Номер формируется автоматически, дата содержит точное время оформления (включая ночные часы).
 * Поле «Магазин» (Реквизит). Тип данных: СправочникСсылка.Магазины. Показывает, через какую площадку пришел заказ.
 * Поле «Товар» (Реквизит). Тип данных: СправочникСсылка.Товары. Указывает на конкретную покупаемую модель бытовой техники.
 * Поле «Количество» (Реквизит). Тип данных: Число, длина 5 знаков, неотрицательное.
 * Поле «ТелефонКлиента» (Реквизит). Тип данных: Строка, длина 20 символов. Используется для связи.
 * Поле «ЗаказПодтвержден» (Реквизит). Тип данных: Булево (значения Истина или Ложь). Отметка диспетчера о совершении звонка клиенту.
 * Свойства: Проведение документа разрешено. Связано с подсистемой «ЗакупкиИПродажи».
7. Документ «Доставка»
Отражает выезд курьера по адресу.
 * Поле «Номер» и «Дата» (Стандартные реквизиты).
 * Поле «ДокументОснование» (Реквизит). Тип данных: ДокументСсылка.ЗаказПокупателя. Связывает доставку со счетом.
 * Поле «АдресДоставки» (Реквизит). Тип данных: Строка, длина неограничена. Заполняется вручную по указанию клиента.
 * Поле «Курьер» (Реквизит). Тип данных: Строка, длина 100 символов. Содержит ФИО ответственного за доставку сотрудника.
 * Свойства: Проведение включено. Связано с подсистемой «ДоставкаТоваров».
### Пункт 4. Определение типов связей между объектами базы данных
Связи в разработанной конфигурации 1С построены на ссылочных типах данных, что автоматически гарантирует ссылочную целостность базы на уровне СУБД.
Связь между справочником «Фирмы» и справочником «Товары» определена как «Один-ко-многим». В реквизитах каждого конкретного товара может быть указана только одна фирма-производитель. При этом в самой базе данных может существовать неограниченное количество товаров, выпускаемых под этой маркой.
Связь между справочником «Магазины» и документом «ЗаказПокупателя» определена как «Один-ко-многим». Один документ заказа жестко привязывается к конкретному интернет-магазину, где была совершена покупка, в то время как сам магазин за время своей работы аккумулирует множество различных документов-заказов.
Связь между справочником «Товары» и документом «ЗаказПокупателя» определена как «Один-ко-многим». В одном документе заказа выбирается одна позиция номенклатуры. В то же время популярный товар может многократно фигурировать в разных документах заказов у разных покупателей за весь отчетный период времени.
Связь между документом «ЗаказПокупателя» и документом «Доставка» определена как «Один-к-одному». По условиям бизнес-логики предприятия, на основании одного оформленного и подтвержденного по телефону заказа может быть создан строго один документ доставки. Создание нескольких доставок на один и тот же заказ, равно как и доставка без существующего заказа, системой блокируется.
